Performance Comparison: Tabu Search vs. Simulated Annealing

Solving the Traveling Salesman Problem

Algorithm Details

TS Tabu Search

Initial Solution: Random arrangement with fixed start/end city

Perturbation: Swap two cities, maintain Tabu list

Neighborhood: Solutions obtained by swapping two cities

Acceptance: Accept solutions not in Tabu list or with better cost

Stopping Criteria: 1000 iterations

Parameters: Tabu tenure = 10

SA Simulated Annealing

Initial Solution: Random arrangement with fixed start/end city

Perturbation: Randomly swap two cities

Neighborhood: All possible small arrangements by swapping

Acceptance: Probabilistic based on cost difference and temperature

Stopping Criteria: Temperature < 1 or 1000 iterations

Parameters: Initial temp = 1000, Cooling rate = 0.995

Results Analysis & Insights

📊 Cost Efficiency

Tabu Search consistently produces lower cost solutions compared to Simulated Annealing, especially as problem size increases.

This is due to its ability to escape local optima using the Tabu list mechanism.

⏱️ Runtime Performance

Simulated Annealing demonstrates faster execution times across all problem instances.

The simpler acceptance criterion and lack of additional data structures contribute to its speed advantage.

📈 Scalability

Tabu Search maintains solution quality as problem size increases, while Simulated Annealing's performance declines.

This indicates Tabu Search is better suited for larger TSP instances.

⚖️ Trade-offs

Tabu Search provides better solutions at the cost of slightly higher runtime.

Simulated Annealing is preferable when computational time is critical and problem size is small.
